What is ATP Greenwich?

ATP Greenwich is a mobile-first PWA for managing an informal neighbourhood tennis league. Players register with invite codes, propose and confirm matches, track standings and ELO ratings, and compete in end-of-season knockout brackets. The admin manages seasons, leagues, group stage fixtures, and bracket seeding through a separate desktop-oriented admin dashboard.

Dev / Prod isolation

When running npm run dev, all database reads and writes are prefixed with _dev/, so local testing never touches production data. Production builds (via npm run build) use paths without a prefix. A yellow "DEV" badge appears in the top bar in dev mode.

S2 — Auth & Onboarding

Phase 1

Invite-only registration

ATP Greenwich is closed — new players must have an invite code generated by the admin. There is no public sign-up page.

  1. Admin generates invite code A random XXXX-YYYY code is saved to invite_codes/ in Firebase. Admin can optionally send it by email via the send-email workflow.
  2. Player navigates to registration page Enters invite code. App validates code exists and is not yet used (used: false).
  3. Player enters details Name, email, password (hashed client-side), and self-assessed skill level. A UID is generated client-side (player_<timestamp>).
  4. Account created in status "invited" Player node is written to players/{uid}. Invite code is marked used.
  5. Admin approves Admin changes status to onboarding in the Players section of the admin dashboard.
  6. Player completes onboarding On first login, they choose an avatar and optionally an alias. Status transitions to active.
  7. Admin assigns player to league Via the Leagues section of the admin dashboard (Player Profile modal or direct league assignment).

Login flow

Player enters email + password. App reads players/ to find a matching email, hashes the entered password, and compares with stored passwordHash. On success, credentials are saved to localStorage (atp_player_creds). No session tokens — all client-side.

Security note: The custom hash function (simpleHash) is not cryptographically secure. Acceptable for a closed, low-stakes app with a small group of known players. Not appropriate for public or commercial use.

Mid-season player move

Admin can move a player between leagues within the same season at any time via the "Move" button in the Leagues section. The player's membership record is removed from the source league and added to the destination league with a transferredAt timestamp.

Group Stage Mechanics

The Group Stage is an optional pre-bracket phase within a league. It runs before the regular-season qualification threshold is reached and gives every player a structured, time-boxed set of fixtures.

How it works

  1. Admin configures group stage Sets matchesPerPlayer (N) and the group-stage deadline (editable at any time). Also configures the five point values (see below).
  2. Admin releases fixtures Clicks "Release Fixtures" — the app generates N randomly-assigned group matches per player (best-effort balanced pairing). Matches are written with groupMatch: true, status: "scheduled", and the league deadline. League groupStageConfig.status transitions to "active".
  3. Players complete matches Each player must play their assigned group matches before the deadline. The Standings tab shows group points as the primary ranking during this phase.
  4. Deadline enforcement The daily cron (scripts/deadline-check.js, run at 08:00 UTC via GitHub Actions) scans all active group matches past their deadline. If both players are absent it sets deadlinePenaltyApplied: true. Points are applied on the fly when standings are calculated.
  5. Admin closes group stage Clicks "Close Group Stage". The app calculates each player's group points, marks qualified: true/false on the member node, and sets groupStageConfig.status to "closed". Players with groupPoints ≥ qualifyPoints (configurable) advance to the bracket.

Point values (all configurable per season/league)

EventDefault pointsConfig key
Played a group match+1pointsConfig.played
Won the group match (bonus)+2pointsConfig.wonBonus
Total for winning a match+3(played + wonBonus)
Pre-arranged match not played before deadline (both absent)-1 eachpointsConfig.missed

Forfeit — opponent (winner)+2pointsConfig.forfeitWinner
The "How scoring works" accordion in the Standings tab reflects the live-configured point values for the current league — players always see the rules that are actually in effect.

Two-match cap: players may play each other at most twice per season/league. The challenge UI hides players already at the cap.

Score entry

Result entry supports up to 3 sets with per-set tiebreak scores. The winner is derived automatically from the sets — the player who won more sets wins the match. A third set (super-tiebreak or full set) is optional.

Photo requirement

A match photo (of the scorecard or handshake) is required to finalise a result. Photos are compressed client-side to max 1280 px JPEG before upload to Firebase Storage. The download URL is stored on the match node and renders as a thumbnail in the Feed.

Disputes

After a result is entered, the other player can dispute it instead of confirming. Disputed matches surface in the admin dashboard with an orange "Disputed" badge. Admin can dismiss the dispute (upholding the original result) or override the score.

Match card fields — Group matches

Group match cards include additional UI elements not present on regular match cards:

ElementDescription
Group badgeBlue pill at the top of the card indicating it is a group-stage fixture
"Play by [date]" labelDeadline label showing the group stage deadline, styled as a secondary label under the opponent name
Forfeit buttonVisible on unfinished group matches — lets the player declare a forfeit (writes forfeited: uid on the match). Only available while the group stage is active.

Admin override

Admin can set any match result directly from the Matches section. This triggers ELO recalculation and marks the match as adminOverride: true.

S6 — Activity Feed

Phase 5

Purpose

The Feed tab is the social hub of the app. It shows a reverse-chronological list of confirmed match results across the whole league, keeping players engaged with what's happening without needing to browse individual profiles.

Feed items

Event typeWhat it shows

Reactions

Players can react to any confirmed match with emoji reactions (tennis ball, fire, trophy, clapping, etc.). Reactions are stored under the match node and aggregated for display. Any player can react; reactions are not namespaced to the current player.

Photo thumbnails

Match result photos render as inline thumbnails in the feed card. Tapping opens the full-size photo in a lightbox overlay.

Tapping any player name or avatar in the feed opens a player history modal showing all confirmed matches, win/loss record, and ELO trend for that player.

S7 — Bracket System

Phase 6

Regular-season qualification

Players qualify for the end-of-season bracket by meeting the league's scoringConfig thresholds: at least minMatches played and at least minWins wins (defaults: 6 and 4). The Standings tab shows a "Qualified" or "Not yet" badge per player.

Group-stage qualification

When the group stage is active, qualification is based on group points: players with groupPoints ≥ qualifyPoints (configured on the league) qualify for the bracket when admin closes the group stage. This overrides regular-season qualification for that league.

Bracket generation

Admin clicks "Generate Bracket" in the admin Bracket section. The app seeds qualified players by standing rank (1 vs N, 2 vs N-1, etc.) into a single-elimination bracket. A bye is assigned for odd player counts. The bracket is saved under seasons/{sid}/leagues/{lid}/bracket.

Bracket progression

Admin sets results for each bracket match. Winners automatically advance to the correct slot in the next round. The bracket reaches a champion when the Final is completed.

Player bracket tab

The Bracket tab in the player app renders the bracket as a visual tree. Players can see their position, opponents, and match results. The bracket is read-only in the player app.

S11 — Notifications

Phase 7

Push notification architecture

Web Push via VAPID. Player opts in via an in-app banner; the browser subscription JSON is saved to players/{uid}/pushSubscription. The scripts/send-push.js script reads all subscriptions and dispatches notifications server-side. Triggered by GitHub Actions workflow dispatch (send-push.yml). The VAPID public key is injected at build time via VITE_VAPID_PUBLIC_KEY; the private key is stored as a GitHub Actions secret.

Notification events

match
Match challenge received — "X has challenged you to a match"
match
Match accepted — "Y accepted your challenge"
result
Result entered — "X entered the result of your match"
result
Result confirmed — "Your match vs X has been confirmed"
group
Group fixture released — "Your group stage fixtures are ready"
group
Match deadline approaching — "Your group match vs X is due in 48h"

Opt-in flow

  1. Service worker registered on app load via public/sw.js.
  2. Banner shown if Notification.permission is "default" and player hasn't dismissed before (push_dismissed localStorage key).
  3. Player taps Enable — browser requests permission. On grant, the app creates a push subscription with the VAPID public key.
  4. Subscription saved to players/{uid}/pushSubscription.
VAPID keys: Run node scripts/generate-vapid.js to generate key pair. Public key goes in .env as VITE_VAPID_PUBLIC_KEY. Private key goes in GitHub Actions secrets as VAPID_PRIVATE_KEY. Keys are not committed to the repo.

S12 — ELO System

Algorithm Standard ELO, K=32

How ELO works

Every player starts at ELO 1000. When a match is confirmed, the winner's and loser's ELO scores are updated using the standard ELO formula with K=32. ELO is global — it persists across seasons and is not reset.

Formula

Expected score: E = 1 / (1 + 10^((Rb - Ra) / 400))

New rating: Ra_new = Ra + K * (S - E)

Where S = 1 for a win, 0 for a loss. K = 32.

ELO tiers

TierELO range
Challenger1200+
Expert1100–1199
Advanced1000–1099
Intermediate900–999
Beginner< 900

ELO history

Each match confirmation appends a { delta, ts, opponent } entry to players/{uid}/eloHistory. The Profile tab renders this as an SVG sparkline, starting from the earliest recorded ELO and ending at the current rating.

Group matches and ELO

Group stage matches follow the same ELO rules as regular matches. A confirmed group match updates ELO exactly like any other confirmed match.
